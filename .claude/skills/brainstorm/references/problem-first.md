# Problem-First Brainstorming

Use this reference when brainstorming starts from a proposed solution, half-baked feature idea, or roadmap full of already-chosen work.

Source: George from prodmgmt.world (@nurijanian), `/problem-first: a simple skill to invert bad ideas`
URL: https://x.com/nurijanian/status/2063186118409929161

## Core Move

Treat every proposed solution as a compressed, imprecise confession of a problem the team senses but has not articulated.

Do not reject the solution immediately. Decompress it back into the problem underneath, then test whether the proposed solution is one of several reasonable responses or a premature jump.

## Triggers

Apply this when the user says things like:

- "We need to build X"
- "Should we add feature Y?"
- "The roadmap already says we need Z"
- "I have too many product ideas"
- "Help me triage this idea"
- "The team wants this solution but I am not convinced"

## Eight-Section Output Pattern

1. **Solution-jumping diagnosis**
   - What signal made this solution feel necessary?
   - What pain, failure, complaint, or opportunity is hiding underneath?

2. **Underlying problem**
   - State the actual user, business, or system problem without naming the proposed solution as the answer.

3. **Assumption challenges**
   - List key assumptions behind the proposed solution.
   - For each: risk if wrong + validation test.

4. **Problem statement**
   - Users/context: who is affected?
   - Struggle: what breaks or slows down?
   - Cause: why does it happen?
   - Consequence: what cost does it create?
   - Success: what would be observably better?

5. **Three alternative framings**
   - Frame A: one interpretation of the problem + matching solution space.
   - Frame B: another interpretation + different solution space.
   - Frame C: third interpretation + different solution space.

6. **Evidence status**
   - None: idea is mostly vibes.
   - Weak: anecdote, one complaint, internal opinion.
   - Medium: repeated user signal, support tickets, product data, interviews.
   - Strong: converging qualitative + quantitative evidence.

7. **Validation plan**
   - What data to inspect.
   - Who to interview.
   - What prototype or experiment to run.
   - What would kill the idea.

8. **Draft stakeholder message**
   - Short note that keeps momentum without blindly accepting the proposed solution.
   - Tone: collaborative, not obstructionist.

## Reverse Use: Idea Triage

For idea-heavy sessions:

1. Take a raw idea.
2. Extract the problem it claims to solve.
3. Assign evidence status.
4. Kill ideas with `Evidence Status: none` unless they are cheap experiments.
5. Promote only ideas with real problem evidence.

## Rules

- Do not start by debating solution feasibility only.
- First ask: what pain would have made a reasonable team propose this?
- Preserve team momentum by using the existing solution as research evidence.
- Challenge assumptions with risk-if-wrong and validation tests.
- Generate at least three alternative problem framings before recommending one build.
- Include a stakeholder message so the user can push back without sounding obstructionist.
- Do not use the framework to delay obvious fixes when evidence is already strong.
